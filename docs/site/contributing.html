Code of Conduct

Our Pledge

We are committed to making participation in Cocapn a harassment-free experience for everyone, regardless of level of experience, gender, gender identity and expression, sexual orientation, disability, personal appearance, body size, race, ethnicity, age, religion, or nationality.

Our Standards

Positive behavior includes:

  • Using welcoming and inclusive language
  • Being respectful of differing viewpoints and experiences
  • Gracefully accepting constructive criticism
  • Focusing on what is best for the community
  • Showing empathy towards other community members

Unacceptable Behavior

Unacceptable behavior includes:

  • Use of sexualized language or imagery
  • Trolling, insulting/derogatory comments, or personal/political attacks
  • Public or private harassment
  • Publishing others' private information without explicit permission
  • Unprofessional conduct

Reporting

To report conduct issues, contact team@superinstance.com. All reports will be reviewed and investigated.

Development Setup

Prerequisites

  • Node.js 18+ — Cocapn runs on Node.js
  • Git — For version control
  • TypeScript — For type-safe development
  • Vitest — Test framework

Fork and Clone

# Fork the repository on GitHub
git clone https://github.com/YOUR_USERNAME/cocapn.git
cd cocapn

Install Dependencies

# Install all package dependencies
npm install

# Install dependencies for local-bridge
cd packages/local-bridge
npm install

# Link for local development
npm link

Run Tests

# Run all tests
npm test

# Run tests for local-bridge
cd packages/local-bridge
npx vitest run

# Run specific test file
npx vitest run tests/bridge.test.ts

# Watch mode for development
npx vitest watch

Type Checking

# Type check all packages
npm run typecheck

# Type check local-bridge
cd packages/local-bridge
npx tsc --noEmit

Testing

Test Status

Cocapn has 196 passing tests on x86_64. 7 tests fail on ARM64 (Jetson) due to age-encryption libsodium WASM — not a code issue.

Writing Tests

Tests are written with Vitest. Follow this pattern:

import { describe, it, expect, beforeEach } from "vitest";

describe("MyComponent", () => {
  beforeEach(() => {
    // Setup
  });

  it("should do something", () => {
    // Arrange
    const input = "test";

    // Act
    const result = myFunction(input);

    // Assert
    expect(result).toBe("expected");
  });
});

Test Categories

  • Unit tests — Test individual functions and classes
  • Integration tests — Test component interactions
  • E2E tests — Test full workflows (bridge startup, chat flow, skills)

Running E2E Tests

# Run E2E tests
npm run test:e2e

# Run specific E2E test
npx vitest run tests/e2e/bridge-startup.test.ts

Commit Guidelines

Commit Message Format

Follow the Conventional Commits specification:

<type>(<scope>): <subject>

<body>

<footer>

Types

  • feat — New feature
  • fix — Bug fix
  • docs — Documentation changes
  • style — Code style changes (formatting)
  • refactor — Code refactoring
  • test — Test changes
  • chore — Build process or auxiliary tool changes

Examples

feat(agents): add multi-agent support

Implement the ability to run multiple agents simultaneously
with routing based on capability and substring matching.

Closes #123
fix(security): resolve secret references at spawn time

Fix bug where "secret:KEY" env values were not dereferenced
at spawn time in agent-spawner.ts.

Fixes #456

Pull Request Process

Before Submitting

  1. Run tests: npm test
  2. Run type check: npm run typecheck
  3. Update documentation if needed
  4. Ensure all commits follow commit guidelines

Creating a PR

  1. Push your branch to your fork
  2. Create a pull request to superinstance/cocapn:main
  3. Fill out the PR template
  4. Link related issues
  5. Request review from maintainers

PR Template

## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
- [ ] Tests added/updated
- [ ] All tests pass

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Documentation updated
- [ ] No merge conflicts

## Issues
Closes #123, #456

Review Process

  • At least one maintainer approval required
  • All CI checks must pass
  • Code must be up-to-date with main branch
  • Review feedback should be addressed

Development Workflow

1. Pick an Issue

Browse issues and find something to work on. Labelled issues include:

  • good first issue — Good for newcomers
  • help wanted — Community contributions welcome
  • enhancement — Feature requests
  • bug — Bug reports

2. Create a Branch

git checkout -b feature/your-feature-name
# or
git checkout -b fix/your-bug-fix

3. Make Changes

# Make your changes
vim packages/local-bridge/src/my-file.ts

# Run tests
npm test

# Commit changes
git add .
git commit -m "feat: add my feature"

4. Push and Create PR

git push origin feature/your-feature-name
# Then create PR on GitHub

Asking Questions

Before asking a question:

  1. Search existing issues and discussions
  2. Read the documentation
  3. Try to reproduce the issue
  4. Gather relevant information (OS, Node version, steps to reproduce)

Reporting Bugs

When reporting bugs, include:

  • Clear title and description
  • Steps to reproduce
  • Expected behavior
  • Actual behavior
  • Environment details (OS, Node version, Cocapn version)
  • Logs and error messages
